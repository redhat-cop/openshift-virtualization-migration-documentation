== Topics

To access content stored on GitLab, your GitLab account must have credentials associated to your account. Two options are available:

. xref:gitlab-credentials[GitLab Credentials]
. xref:bitwarden[Bitwarden]
. xref:ssh-key-generation[SSH Key Generation]

The following sections will describe how to configure your GitLab account with these credential types.

Login to link:https://gitlab.consulting.redhat.com[GitLab] and navigate to the _SSH Keys_ section of your User Setting by clicking link:https://gitlab.consulting.redhat.com/-/user_settings/ssh_keys[here].

Click the *Add new key* button to begin the configuration process.

Paste the contents of the SSH Public Key previously copied in the _Key_ text field. Include an optional _Title_ to distinguish the key from other keys, select the desired _Usage type_ (The default option is satisfactory, however, only _Authentication_ is required) and optionally select the date the key will expire.

Click the *Add key* button to add the SSH Public Key to your account. A confirmation page will be displayed indicating the values that were configured.

To verify that you can successfully communicate with GitLab using the SSH key configured in your account, execute the following command.

[source,shell]
----
ssh -p 2222 -T git@git.consulting.redhat.com
----

If this is the first time you are connecting to GitLab, you will receive a warning indicating that you need to verify the authenticity of the host. Type `yes` and hit *Enter*.

The following message will be displayed indicating a successful connection to the GitLab server.

[source,text]
----
Welcome to GitLab, @jdoe!
----

=== Personal Access Token

A GitLab link:https://docs.gitlab.com/user/profile/personal_access_tokens[Personal Access Token (PAT)] can be used instead of SSH keys to authenticate to GitLab.

Login to link:https://gitlab.consulting.redhat.com[GitLab] and navigate to the _Personal Access Token_ section of your User Setting by clicking link:https://gitlab.consulting.redhat.com/-/user_settings/personal_access_tokens[here].

Click the *Add new token* button to begin the configuration process.

Enter the name for the Personal Access Token (such as `openshift-virtualization-migration) and optionally enter a _Token Description_ and _Expiration Date_.

Under the _Select scopes_ section, select *read_repository*.

Click then *Create personal access token* to create the token.

A confirmation page will be displayed and includes the value of the generated token. Be sure to copy the value as it cannot be accessed after the page is closed.

GitLab repositories can now be accessed using the _HTTPS_ protocol.

== Bitwarden

link:https://bitwarden.com[Bitwarden] is used to store sensitive resources for the Ansible for OpenShift Virtualization Migration Project and are located in a Bitwarden link:https://bitwarden.com/help/about-collections[Collection]. To gain access to the collection, you must have a Bitwarden account associated with your @redhat.com Email Address.

Use the following steps to complete the setup process.

. link:https://bitwarden.com/go/start-free[Create a Bitwarden account] using your Red Hat email address
. Join the `migration-factory` Google Group. The easiest way to join the Google Group is to onboard to the Ansible for OpenShift Virtualization Migration using link:https://docs.google.com/document/d/1zyx4vl6evt15jgHBcpaGOfcjcU6CSqUMLqQVBSGmdoE[this] form.
. Download one of the supported applications or use the Bitwarden website to access the resources that are available to you.

Once the membership to the _migration-factory_ Google Group has been confirmed, it may take some time until the Bitwarden Collection called _Migration Factory_ is available.

When the synchronization has completed, the _Migration Factory_ Collection should be present under the _Collection_ section of the Bitwarden Vault.

== SSH Key Generation

An SSH Key can be used to facilitate the communication between a local machine or the Ansible for OpenShift Virtualization Migration to external tools. The `ssh-keygen` command is used to create an SSH keypair for this purpose.

Create a new SSH keypair of type `rsa`. An example of the steps for generating the keypair ia shown below. Be sure to update the email address associated with the `-C` parameter with your actual email address:
